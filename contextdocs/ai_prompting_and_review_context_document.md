# AI Prompting and Review Context Document

## Project Overview

This document defines how AI-generated code is prompted, reviewed, tested, and approved before merging into the repository for the **Candidate-Facing Applicant Tracking System (ATS)**. The system helps job seekers organize and manage their job search activities and is treated as a production-quality internal application.

The goal of this policy is to ensure that AI-assisted development improves productivity **without compromising correctness, security, maintainability, or team understanding of the codebase**.

---

## Technical Stack Context

Developers must ensure AI-generated code aligns with the project's architecture:

Framework: Next.js 15 (App Router)
Language: TypeScript (strict mode)
Database: PostgreSQL (Supabase)
ORM: Prisma
Authentication & Storage: Supabase
AI Integration: Google Gemini API
Styling: Tailwind CSS
Deployment: Vercel
CI/CD: GitHub Actions

All AI-generated outputs must respect these constraints.

---

## Approved AI Usage Scope

Developers may use any AI tooling (e.g., ChatGPT, Copilot, Gemini, Claude, Cursor), provided that:

- Only sanitized code snippets are shared externally
- No secrets are included
- No database credentials are exposed
- No environment variables are shared
- No proprietary logic is copied verbatim into external prompts

Sanitized snippets include:

- isolated utility functions
- UI components
- schema fragments
- pseudocode logic

Non-sanitized content must never be shared.

---

## Prompting Standards

When requesting AI-generated code, developers must provide structured prompts that include:

### Required Prompt Components

1. Objective
   Example:
   "Create a reusable Next.js server action that inserts a job application record using Prisma."

2. Constraints

Always specify:

- Next.js App Router compatibility
- TypeScript strict mode
- Prisma usage (not raw SQL)
- Tailwind styling conventions (if UI-related)
- Supabase auth compatibility (if user data involved)

Example:
"Use Prisma ORM and ensure strict typing. Do not use client-side database calls."

3. Expected Input/Output Types
   Example:
   "Input: application form data
   Output: Prisma create record response"

4. Error Handling Expectations
   Example:
   "Include validation and safe error handling suitable for production"

5. Testing Expectations
   Example:
   "Return code compatible with Jest/Vitest unit testing"

6. Security Constraints
   Example:
   "Assume user authentication is required before insertion"

---

## Example High-Quality Prompt Template

Use this template when prompting AI:

"Generate a TypeScript server action for Next.js 15 App Router that inserts a job application record into PostgreSQL using Prisma. The function should:

- run server-side only
- validate inputs
- assume Supabase authentication already verified
- follow strict typing
- return structured error responses
- support unit testing
- avoid raw SQL

Return production-quality code only."

---

## AI-Generated Code Identification Requirements

All AI-generated code must be disclosed inside the pull request description.

Include:

- the prompt used
- the AI tool used
- what modifications were applied after generation

Example PR note:

"Initial draft generated using Gemini. Refactored manually for Prisma typing and validation."

---

## Developer Responsibility Rule

AI-generated code must never be merged without the author fully understanding:

- what the code does
- why it works
- how errors are handled
- how it interacts with authentication
- how it affects database writes

If a developer cannot explain the code, it must not be merged.

---

## Mandatory Testing Requirements

All AI-generated code must pass the following before merge:

### Unit Tests (Required)

Must include tests for:

- expected success behavior
- invalid input handling
- authentication edge cases (if applicable)
- database write/read correctness

### Linting (Required)

Code must pass:

npm run lint

### Type Safety (Required)

Code must pass:

TypeScript strict mode validation

---

## Pull Request Review Requirements

AI-generated contributions require:

- standard PR review process
- minimum of two reviewers
- at least one reviewer verifying correctness of logic

Reviewers must confirm:

- architecture alignment
- security compliance
- readability
- maintainability
- typing correctness

---

## Security Review Expectations

Because this system manages job-search-related user data, reviewers must confirm:

- no credential exposure
- no client-side database writes
- no unsafe API usage
- correct authentication assumptions
- safe error responses

If database schema changes are generated by AI, reviewers must validate Prisma migrations manually.

---

## Gemini API Usage Policy

AI-assisted features inside the application must:

- clearly document prompt structure
- avoid storing raw prompts containing sensitive data
- validate responses before persistence
- handle hallucinated outputs safely

Gemini responses must never be trusted without validation.

---

## Database Safety Rules for AI Code

AI-generated database logic must:

- use Prisma exclusively
- avoid raw SQL unless reviewed manually
- validate schema compatibility
- handle null cases safely
- avoid unsafe cascading writes

---

## Frontend Safety Rules

AI-generated UI components must:

- follow Tailwind styling conventions
- remain accessible
- avoid unnecessary client components
- respect server/client boundary rules in App Router

---

## CI/CD Enforcement

GitHub Actions automatically enforce:

- linting checks
- type validation
- test execution

PRs failing checks cannot be merged.

---

## Approval Requirements

Before merge:

- minimum two reviewers must approve
- author must confirm understanding of generated code
- lint must pass
- unit tests must pass
- TypeScript strict checks must pass

Any reviewer may approve once requirements are satisfied.

---

## When AI Should NOT Be Used

Avoid AI assistance when writing:

- authentication flow logic
- schema migration logic without review
- security-sensitive middleware
- environment configuration

These areas require manual implementation or senior verification.

---

## Acceptable AI Use Cases

AI is encouraged for:

- component scaffolding
- utility helpers
- validation schemas
- Prisma query drafting
- unit test generation
- documentation writing

---

## Documentation Requirements

If AI-generated code introduces:

- new helpers
- schema changes
- API routes
- shared utilities

Developers must update documentation before merge.

---

## Team Agreement Summary

AI tools accelerate development but do not replace engineering responsibility.

All contributors agree that:

- generated code must be understood
- generated code must be reviewed
- generated code must be tested
- generated code must meet production standards

Following this document ensures AI improves delivery speed while maintaining reliability and trust in the codebase.
